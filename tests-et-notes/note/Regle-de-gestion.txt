1 Contexte général
       Une entité CentreCommercial sera créée manuellement et à l’avance dans la base de données.


  CentreCommercial {
     nom:            String, required
     description: String, optional
     adresse:        String, required,

     telephone: String, optional,
     photo: String, optional,
     createdAt:      Date, default

 }


       L’application comporte 3 types d’utilisateurs, correspondant à 3 rôles :

                Admin

                Acheteur

3.2 PorteFeuille

Entité représentant le porte-monnaie virtuel d’un utilisateur (Admin, Commercant ou Acheteur) (facilitera l'integration 'une systeme de paiement en ligne reelle donc ce
sera utile pour une bonne conception).

Elle permet de suivre les transactions :

       Location
       Achat
       Commission


Cas du paiement de loyer (type = "Loyer")

       Une transaction avec le PorteFeuille de l’Admin dans "toWallet" => + l'amount sur la balance de son PorteFeuille
       Une transaction avec le meme amount et le PorteFeuille du Commercant dans "fromWallet" => - l'amount sur la balance de son PorteFeuille


Cas d’un achat (type = "Achat")

       Une transaction avec le PorteFeuille du Commercant dans "toWallet"
       Une transaction avec le PorteFeuille de l’Acheteur dans "fromWallet"


  PorteFeuille {
     owner:        ObjectId, required, ref "User"
     balance:      Number, required, default 0
     createdAt: Date, default
     updatedAt: Date, default
 }
 PFTransaction {
     fromWallet: ObjectId, required, ref "PorteFeuille" // celui qui paie
     toWallet:      ObjectId, required, ref "PorteFeuille" // celui qui reçoit
     type:          String, required, enum ["Achat", "Loyer", "Commission"]
     amount:        Number, required // toujours positif
     description:String, optional
     createdAt:     Date, default
 }


        type : "Achat" , "Loyer" , "Commission"

II - COMMERCANT
Règles de gestion

       Le Commercant s’inscrit via une interface classique.

       Après connexion, il peut :

                Créer une ou plusieurs boutiques
                Voir la liste de ses boutiques

Pour une boutique existante, il peut :

       Modifier les informations de la boutique

       Faire une demande de location d’espace

       Fermer totalement la boutique (statut = "Inactif" )

       Voir l’historique de paiement des loyers
          Payer le loyer mensuel → création d’un Recepisse (Reçu) avec :

                     receveur = Commercant
                     donneur = Admin

          Gérer :

                     Types de produits (de ce boutique)
                     Les Produits
                     Le stock dans chaque produit (Modifier)

          Gérer les achats :

                     Achats en cours (Livraison / Récupération)
                     États : "EnAttente" , "Validee" , "Annulee"
                     Saisie de la durée de livraison
                     Historique des achats




Explication:

 _ Lorsqu'un commercant entre dans une de ses boutiques dans la liste deja existant, il pourra:
          _ modifier les informations d'une boutique
          _ effectuer une demande de location d'un espace parmis les espaces disponible
          _ Interface permettant de fermer totalement la boutique => Son etat devient "Inactif"
          _ voir l'historique de paiement des loyers et aura une interface de paiement du loyer mensuelle
                   Une entité Recepisse(Recu) sera creer avec le receveur (Commercant) et le donneur(l'Admin)
          _ ajoute/modifier/supprimer les types de produits que sa boutique vend (ex: Food, Smartphone, Bijout, ...)
          _ ajouter/modifier/supprimer des produits (donc une liste de produits)
                   Chaque nouveau produit aura un stock (detient le nombreDispo d'un produit, avec 0 par defaut) lié à lui,
                   Un produit a un temps de preparation ou non (donc une temps hh:mm:ss ou null) selon le produit, null signifie disponible de
                   suite
          _ gerer l'information sur le stock d'un produit => modification du nombreDispo des produits
          _ voir la liste des achats en cours, séparé par 2 categories: Livraison et Recuperation (voir la regle en bas dans Acheteur):
                   Chaque achat en cours auront un etat parmis "EnAttente", "Validee", "Annulee"
          _ saisir la duree de livraison d'un achat a "Livrer", l'etat de l'achat "EnAttente" devient => "Validee"
          _ Historique des achats

III - ACHETEUR
Règles de gestion

          L’Acheteur s’inscrit via une interface classique.

          Après connexion, il peut :

                 Voir la liste paginée des boutiques
                 Voir la liste des produits d’une boutique
                 Faire une recherche d'une boutique avec un mots clé: nom de la boutique ou le nom d'un Commercant
                 Voir sa liste de ses achats en cours
                 Voir son historique d'achat
                 Voir la liste de ses factures (pour recuperer son recu en pdf coté frontend)


Achat

          Un Achat est lié a une TypeAchat, dont 2 types :

                  "Recuperer"
                  "Livrer"

          son TypeAchat contient :

                  dateDebut
                  dateFin donc si dateDebut = dateFin => disponible de suite

Règles :
       Récupérer :

                  dateFin = dateDebut + tempDePreparation donc si tempDePreparation = 0 => disponible de suite

       Livrer :

                  L’achat est en "EnAttente"
                  Le Commercant saisit une durée de livraison
                  dateFin = dateDebut + durée
                  L’état devient "Validee"


Panier

       Objet Panier (DTO, non persisté)

       Regroupe les achats (dans le frontend) avant validation(envoye des listes d'achat avec appel api)

       Lors de la validation :

                  Les Achats sont envoyés au backend
                  Une Facture est créée, et cette facture sera associés a chaque Achat pour pouvoir les suivres/regrouper
                  L'acheteur pourra plutard recevoir un Reçu généré pour les achats validés (etat "Validee") grace a ce facture

       À l’insertion d’un Achat "EnAttente" ou "Validee":

                  Le stock dans le Produit est vérifié (s'il y en a encore de nombreDispo)
                  Le nombreDispo du stock est décrémenté

       Lorsqu’un Achat est "Validee" :

                  Transactions PorteFeuille Acheteur ↔ Commercant
